# Fitness Agent MVP 中文版

## 目标

创建一个本地优先的 AI 减脂健身助手。它可以记录每天的饮食、体重和简单运动数据，并总结热量摄入、估算消耗以及用户目标进展。

第一版要先跑通核心闭环：

```text
用户描述数据
  -> Agent 或 CLI 转成结构化输入
  -> Core service 校验并记录
  -> SQLite 本地保存
  -> Summary service 返回当天汇总
```

## 目标用户

MVP 面向单个本地用户。第一版不做多用户、不做云同步、不做账号系统。

## 第一版范围

### 用户档案

- 创建或更新本地用户档案。
- 记录身高、体重、年龄、性别、活动水平和目标体重。
- 基于档案估算 BMR 和 TDEE。
- 在条件足够时记录目标热量和蛋白质目标。

### 饮食记录

- 按日期和餐次记录饮食。
- 记录食物名称、数量、单位、估算克重、热量、蛋白质、碳水和脂肪。
- 支持用户手动提供营养值。
- 缺少精确数据时允许估算。
- 返回单餐和全天总计。

### 体重记录

- 按日期记录体重。
- 返回最近体重。
- 数据足够时返回简单 7 日均重。

### 运动和训练记录

- 记录简单活动，比如走路、跑步、骑车、力量训练。
- 记录时长和估算消耗热量。
- MVP 阶段力量训练可以先用自由文本或简单结构化记录。

### 每日总结

- 展示某一天的总摄入热量。
- 展示某一天的估算运动消耗。
- 如果用户档案和目标数据足够，展示目标热量。
- 展示相对目标的剩余热量。
- 展示蛋白质摄入和蛋白质目标。
- 明确标注估算值。

### CLI

- 提供 Python CLI，方便本地使用和调试。
- 初始命令覆盖用户档案、饮食记录、体重记录、活动记录和每日总结。

### MCP 工具

- CLI 和 core services 跑通后，再暴露 MCP tools。
- 初始 MCP tools：
  - `record_meal`
  - `record_weight`
  - `record_activity`
  - `get_daily_summary`
  - `get_user_profile`
  - `update_user_profile`

### Skill

- MCP tools 存在后，再创建 Codex/Agent Skill。
- Skill 负责说明：什么时候记录数据、什么时候追问、什么时候估算、如何表达不确定性。

## 暂不做

- Web UI。
- 移动 App。
- 多用户账号。
- 登录认证。
- 云同步。
- 支付或订阅。
- 完整训练计划引擎。
- 完整 RAG 知识库。
- 模型微调或蒸馏。
- 自动识别食物照片。
- 可穿戴设备接入。
- 公开插件发布。

## 第一里程碑

满足以下条件，第一里程碑完成：

```text
1. 初始化用户档案。
2. 记录一顿饭。
3. 记录一次体重。
4. 记录一次活动。
5. 通过 CLI 查询当天总结。
6. 数据持久化到 SQLite。
7. 核心 summary 逻辑有测试。
```

## 第二里程碑

满足以下条件，第二里程碑完成：

```text
1. MCP server 可以本地启动。
2. MCP tools 调用和 CLI 相同的 core services。
3. Agent 可以通过 MCP 记录饮食。
4. Agent 可以通过 MCP 查询每日总结。
5. 有基础 Skill 说明。
```

## 第三里程碑

满足以下条件，第三里程碑完成：

```text
1. 可以通过 CLI 和 MCP 记录体重。
2. 可以返回最近体重，数据足够时返回简单 7 日均重。
3. 可以通过 CLI 和 MCP 记录活动。
4. 每日总结包含活动消耗和净热量。
5. Skill 工具契约文档包含体重和活动 tools。
```

## 数据质量原则

- 进入 core services 的数据尽量结构化。
- 必要时保留原始用户文本，方便后续追溯。
- 记录数值来源：用户提供、数据库查询、系统估算。
- 不隐藏不确定性，估算值要在总结里明确展示。

## 安全原则

- 不做医疗诊断。
- 不提供危险减重建议。
- 对症状、受伤风险、进食障碍信号或医学情况，建议咨询专业人士。
- 建议应保守、实用、可执行。

## 后续方向

MVP 跑通后再考虑：

- 接入食物营养数据源。
- 增加超过简单 7 日均重之外的周趋势分析。
- 增加结构化力量训练记录和肌群映射。
- 增加营养与训练知识库 RAG。
- 打包成可安装插件。
- 如果本地助手验证有价值，再加 Web 或移动端界面。
